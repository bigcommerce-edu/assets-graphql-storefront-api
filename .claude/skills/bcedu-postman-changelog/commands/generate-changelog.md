**When to apply:** Generating a Markdown changelog by comparing two versions of a Postman collection JSON file.

Your job is to compare two versions of a Postman collection and produce a structured Markdown changelog file saved to a `CHANGELOGS/` directory in the project root.

---

## Step 1 — Determine Input Mode

Identify which mode applies based on the user's request:

- **Two-file mode**: The user explicitly provides two file paths (old and new). Use those directly.
- **Staged/uncommitted mode**: Default when no comparison target is specified; compare the working tree file against `HEAD`.
- **Branch vs. main mode**: The user says "compared to main" or similar; compare `main`'s version against the current working file.

Ask for clarification if the mode is ambiguous.

---

## Step 2 — Locate the Collection File

If the user did not specify a file path, search the project root:

```bash
find . -maxdepth 2 -name "*.postman_collection.json" | head -5
```

If multiple files are found, ask the user which one to use. If none are found, ask the user to provide the file path.

---

## Step 3 — Extract Both JSON Versions

Write both versions to temp files for reliable parsing:

```bash
# Branch vs. main mode
git show main:"<path>" > /tmp/old_collection.json
cp "<path>" /tmp/new_collection.json

# Staged/uncommitted mode
git show HEAD:"<path>" > /tmp/old_collection.json
cp "<path>" /tmp/new_collection.json

# Two-file mode
cp "<old-path>" /tmp/old_collection.json
cp "<new-path>" /tmp/new_collection.json
```

If `git show` fails (the file did not exist in that ref), write an empty JSON object to the old version:

```bash
echo '{}' > /tmp/old_collection.json
```

---

## Step 4 — Parse and Compare with Python

Run the skill's diff script:

```bash
python3 .claude/skills/bcedu-postman-changelog/scripts/diff_collections.py
```

The script reads `/tmp/old_collection.json` and `/tmp/new_collection.json` and prints structured diff output to stdout. Capture this output — you will use it to build the changelog in Step 7.

---

## Step 5 — Determine the Comparison Description

Based on the input mode, choose a description for the changelog header:

- Two-file mode: `Changes between <old-file-path> and <new-file-path>`
- Staged/uncommitted mode: `Changes from last commit (HEAD)`
- Branch vs. main mode: `Changes from main branch`

---

## Step 6 — Check for Naming Conflicts

The output file goes in `CHANGELOGS/` at the project root. The default name is `CHANGELOG-Postman-YYYY-MM-DD.md` using today's date.

Before writing:
1. Check if `CHANGELOGS/` exists. Note that it may need to be created.
2. Check if `CHANGELOGS/CHANGELOG-Postman-YYYY-MM-DD.md` already exists.
3. If it does, inform the user and ask whether to overwrite it or use an alternate name (e.g., `CHANGELOG-Postman-YYYY-MM-DD-2.md`). Wait for confirmation before proceeding.

---

## Step 7 — Build and Write the Changelog

Use the diff output from Step 4 to construct the Markdown content. Follow this structure:

### Header

```markdown
# Changelog: <Collection Name>

**Date:** YYYY-MM-DD
**Comparison:** <description from Step 5>
**Collection file:** `<relative path to collection file>`
```

### Global Changes

Only include sub-sections that have actual changes. Omit sub-sections with no changes.

```markdown
## Global Changes

### Collection Variables
- **Added:** `key` (initial value: `value`)
- **Removed:** `key` (was: `value`)
- **Modified:** `key`: `old value` → `new value`

### Collection Auth
(describe the change in plain language)

### Collection-Level Event Scripts
(list which event types changed — prerequest, test — and briefly describe the change)
```

### Per-Section Changes

Group folders by base name — strip the ` (Automated)` suffix to find the base. One section covers both the standard folder and its automated counterpart.

```markdown
## <Base Folder Name>
> Covers both the "<Base Folder Name>" and "<Base Folder Name> (Automated)" folders.

### Added Requests
- `Request Name` — added to [standard / automated / both]

### Removed Requests
- `Request Name` — removed from [standard / automated / both]

### Modified Requests

#### `Request Name`
- **GraphQL query updated** *(both)*
- **Test scripts updated** *(automated only)* — briefly describe what changed if discernible
- **URL changed** *(standard only)*
- **Headers changed** *(both)*
- **Raw body changed** *(both)*
```

**Rules:**
- Omit any section or sub-section with no changes.
- When a change appears in both standard and automated folders, describe it once and note "both".
- When a change only affects one folder, call it out explicitly: "automated only" or "standard only".
- If a folder has no automated counterpart, omit the blockquote note.
- If no differences are found at all, write a "No changes detected between the two versions" statement after the header. Still write the file.

### Footer

```markdown
---
*Generated by Claude Code on YYYY-MM-DD*
```

### Write the File

```bash
mkdir -p CHANGELOGS
```

Write the complete Markdown content to `CHANGELOGS/CHANGELOG-YYYY-MM-DD.md` (or the confirmed alternate name). Confirm the output path to the user.

---

## Edge Cases

- **Malformed JSON**: Report the parse error with the file path and stop.
- **File not in git ref**: Treat old version as `{}` — all current content is described as "added".
- **Collections with sub-folders deeper than 2 levels**: Process top-level folders as sections; note the deeper nesting in the changelog header.
- **Request renames**: No rename detection — treat as a remove + add. The user may annotate manually if needed.
